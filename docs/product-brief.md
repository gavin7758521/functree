# FuncTree 产品说明

FuncTree 是面向 AI 协作的软件功能知识库服务端。

它不再要求用户在代码仓库里维护本地配置目录，也不再以本地文本文件作为核心数据源。FuncTree 的主形态是一个后端服务：数据存储在数据库中，AI 工具通过 MCP 写入和查询，人类通过中文 Web 管理台审查和维护。

## 产品定位

FuncTree 解决的问题是：AI 工具可以快速读写代码，但团队缺少一个稳定的“功能层知识库”来表达产品、需求、前端、后端、UI/UX、测试、文档之间的对应关系。

FuncTree 不是任务管理系统，也不是代码扫描器。它关注的是：

- 一个项目里有哪些功能集。
- 每个功能集有哪些功能。
- 功能是否有多个版本共存。
- 不同角色、不同版本、不同粒度的功能之间如何对齐。
- AI 工具如何可靠地把分析结果同步给团队。

## 核心层级

```text
项目 Project
  功能集 Feature Set
    功能 Feature
      子功能 Feature
```

### 项目

项目是平台内的最高层级，代表一个产品、系统、业务线或长期工作空间。项目表达当前知识库的最新状态。

项目字段包括：

- 专属 ID
- 名称
- 当前版本
- 状态：活跃、暂停、归档
- 描述
- 元数据

### 功能集

功能集属于项目。每个项目可以有多个功能集，功能集可以多版本共存。

功能集适合表达不同角色或不同交付视角，例如：

- 前端功能集 v2.1
- 后端功能集 v1.5
- 产品需求 2026.07
- UI/UX 交互稿
- 测试功能集
- 运维功能集

功能集字段包括：

- 专属 ID
- 项目 ID
- 名称
- 版本
- 类型：前端、后端、产品、UI/UX、需求、测试、文档、运维、其他
- 状态：正常、草稿、冻结、归档、废弃
- 负责人
- 描述

### 功能

功能属于功能集。功能可以包含子功能，也可以有多个版本共存。

功能字段包括：

- 专属 ID
- 稳定键 stableKey
- 名称
- 版本
- 状态：草稿、进行中、待评审、已完成、已上线、已归档、已废弃、阻塞中
- 类型：能力、模块、页面、接口、组件、流程、规则、测试、文档、其他
- 父功能 ID
- 描述
- 元数据

稳定键用于表达长期语义，例如 `login`；功能 ID 表达某一个具体版本或实例，例如 `feat_app_login_v21`。

## 对齐关系

对齐关系是 FuncTree 的核心机制。

对齐不是层级关系，也不限制同级别。项目、功能集、功能都可以作为可对齐对象。一个对齐关系可以连接两个或多个对象，用来表达这些对象在产品语义、实现目标、交付范围、依赖责任或验证标准上存在对应关系。

例如：

```text
产品功能：用户登录
App 前端功能：登录页面
后端功能：登录接口
测试功能：登录链路用例
```

它们可以组成一个“登录链路对齐关系”。

对齐关系类型包括：

- 对应
- 实现
- 支撑
- 验证
- 依赖
- 替代
- 冲突
- 覆盖
- 拆解
- 关联

## 服务端优先

FuncTree 第一版使用：

- Node.js + TypeScript
- SQLite 作为默认数据库
- Fastify 提供 HTTP API
- 独立 npm 包 `@gavin7758521/functree-mcp` 提供 stdio MCP 远程适配器
- React + Vite 提供中文 Web 管理台

SQLite 是默认存储介质，因为它部署简单、事务可靠、适合本地和小团队使用，并且支持 FTS5 全文搜索。未来可以增加 Postgres 适配器。

## AI 工作流

推荐工作流：

1. 用户创建项目。
2. 用户或 AI 创建功能集。
3. AI 读取代码、需求、设计或测试信息。
4. AI 通过 MCP 写入功能、子功能和版本信息。
5. AI 或用户建立跨层级对齐关系。
6. 用户在中文 Web 管理台审查功能知识库。

## 不做的事情

第一版不做：

- 仓库内本地配置目录作为主存储。
- 文本文件 schema 作为主数据模型。
- 文件工具式渲染和校验流程。
- 用户账号和云同步。
- 权限系统。
- 自动代码扫描并声称完全理解项目。

FuncTree 第一版的目标是：先把服务端数据模型、MCP 工具入口和中文管理台做稳。
